# AG4-3 建筑平面布局评价

> 模块版本：v1.5（对齐 Scorer v2.0 实现）

---

## 一、模块目标

接收 AG4-2 生成的多个布局方案，对每个方案进行量化评价。评价体系不仅关注物理约束（MUST 邻接），还引入了空间有效率、便捷性、形状利用率等多个维度的量化指标，最终输出包含**综合评分**、**多样性惩罚**及**详细得分明细**的评价结果，供用户选择最优方案。

---

## 二、在系统中的位置

```
AG4-2（建筑平面布局生成，见 AG4建筑平面布局3_泵站业务.md）
  │  输出：LayoutVariant[]（含 placements, violations, adjacency）
  ▼
AG4-3（建筑平面布局评价）← 本模块 (src/layout/scorer.js)
  │  输出：EvaluatedVariant[]（含 score, breakdown, variantType）
  ▼
UI 展示（src/ui/layout-panel.js，展示排名与得分明细）
```

## 三、评价指标体系（Breakdown 详解）

评分采用**基础分 + 奖励分 - 惩罚分**的累计方式。为了引导算法生成高质量方案，各指标被划分为三个主次层级：

### 第一梯队：硬性功能红线（Hard Redlines）
**目标**：确保方案在功能上合法且具备落地可能性。此类惩罚具有“一票否决”的效果。

| 指标字段 | 建议分值 | 目的与逻辑 |
| :--- | :--- | :--- |
| **缺失房间 (`missingRooms`)** | **−500 pt / 间** | 最严重错误。设备无处安放，方案直接视为无效。 |
| **可达性违规 (`doorAccess`)** | **−300 pt / 间** | 物理隔绝。如服务间不靠外墙（无法对外开门）、设备间不靠走廊，导致人员或设备无法进入。 |
| **MUST 约束违反 (`violations`)** | **−200 pt / 项** | 违反明确的业务红线（如变压器室未相邻）。每项未满足的 MUST 关系均扣分。 |

---

### 第二梯队：空间质量生命线（Spatial Quality）
**目标**：在方案合法的前提下，追求房间形状的规整度与建筑效率，拉开质量差距。

| 指标字段 | 建议分值 | 目的与逻辑 |
| :--- | :--- | :--- |
| **形状与比例 (`aspectRatio`)** | **−100 pt / 单位违规** | 抑制畸形。含三项检查：1. 长宽比超标（>4:1）；2. 利用率低（<70%）；3. 顶点数 > 6。每单位违规叠加扣分。 |
| **空间有效率 (`efficiency`)** | **最高 +150 pt** | 衡量建筑紧凑度。排除走廊/吊装口后的功能空间占比，越高分值越高（线性映射 0.60~0.90）。 |
| **走廊完整性 (`corridor`)** | **最高 +80 pt（线性）** | 鼓励形成清晰的交通枢纽。走廊连接的房间数越多，奖励越高（线性比例映射，以 `corridorHitsThreshold` 为满分基准，超过上限仍封顶）。走廊生长阶段同时偏好细长形态，以自然满足 ≥1500mm 宽度要求。 |

---

### 第三梯队：锦上添花的优化项（Optimization）
**目标**：在多个优质方案中微调位次，寻找流线与成本的最佳平衡。

| 指标字段 | 建议分值 | 目的与逻辑 |
| :--- | :--- | :--- |
| **MUST 临近满足 (`adjacency`)** | **+40 pt / 对** | 虽然违反会重罚，但满足 MUST 关系依然提供基础正向分。 |
| **SHOULD 临近满足 (`adjacency`)** | **+20 pt / 对** | 建议性的功能靠近（如风机房靠走廊、服务用房集中）。 |
| **便捷性 (`accessibility`)** | **最高 +40 pt** | 缩减入口至各房间的平均步行距离。当前实现仅计算地面层（入口取建筑南侧中点），一层便捷性**暂忽略**，待后续版本补充。 |
| **变压器布置 (`trafo`)** | **最高 +50 pt** | 奖励变压器靠外墙（+20/台）或同侧（+20）布置。 |
| **风机房距离 (`fanRoom`)** | **最高 +30 pt** | 奖励风机房靠近一层吊装口（距离线性衰减）。 |
| **占地面积 (`footprint`)** | **−8 pt / m²** | 控制总体量。作为慢变量，不应压倒空间质量（注：当前暂由用户指定尺寸生成）。 |

## 四、输出结构定义

```ts
interface EvaluatedVariant {
  score: number              // 最终总得分
  spaceEfficiency: number    // 空间有效率原始值 (0~1)
  efficiencyScore: number    // 空间有效率对应的奖励分
  accessibilityScore: number // 便捷性对应的奖励分
  aspectRatio: number        // 方案中所有房间的最大长宽比
  diversityPenalty: number   // 相似度惩罚（由 mergeVariants 在排名后统一应用）
  
  breakdown: {
    base: number             // 基础分 10000
    footprint: number        // 占地面积扣分 (≤0)
    trafo: number            // 变压器奖励 (≥0)
    fanRoom: number          // 风机房奖励 (≥0)
    adjacency: number        // 邻接满足奖励 (≥0)
    adjMustCount: number     // 满足 MUST 的数量
    adjShouldCount: number   // 满足 SHOULD 的数量
    corridor: number         // 走廊奖励 (≥0)
    efficiency: number       // 空间效率奖励 (≥0)
    accessibility: number    // 便捷性奖励 (≥0)
    violations: number       // MUST 违反扣分 (≤0)
    violationCount: number   // MUST 违反项数
    violationDetails: string[] // 违反项详情（房间ID/对）
    doorAccess: number       // 可达性违规扣分 (≤0)
    doorAccessCount: number  // 可达性违规房间数
    missingRooms: number     // 房间缺失扣分 (≤0)
    missingRoomCount: number // 缺失房间数量
    aspectRatio: number      // 形状与比例扣分 (≤0)
    aspectRatioCount: number // 形状违规总次数（含利用率低及顶点过多）
  }
}
```
