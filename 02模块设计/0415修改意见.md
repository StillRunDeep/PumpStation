# AG4 建筑平面布局模块修改意见

> 日期：2026-04-15  
> 覆盖文档：AG4建筑平面布局1~4

---

## 一、模块架构层面

### 1.1 模块边界不清晰：生成与评价耦合

文档3（泵站业务）的 Step 7 直接调用评价函数，但评价文档（文档4）里 `variantType` 分类（`compact / large-repair / large-parking / standard`）的分类逻辑未记录在任何文档中。建议：
- 在文档4补充 `variantType` 的分类规则（用什么阈值、哪个字段决定类型）
- 将评价逻辑完全收敛到 `scorer.js`，生成器不应感知评分细节

【意见回复】：
- 文档4删除`variantType` 分类（`compact / large-repair / large-parking / standard`）的分类逻辑
- 将评价逻辑完全收敛到 `scorer.js`，生成器允许调用评分的某个接口，作为下一步参考

### 1.2 走廊宽度约束缺乏执行路径

文档3规定走廊净宽 ≥ 1500mm，但约束生长算法（文档1）里走廊参与面积补偿时是等比放大的。当建筑轮廓较大时，走廊可能被生长成一个宽房间而非线性通道，**宽度约束实际上没有被强制执行**。建议：
- 走廊应有"线性约束"：在 Step 3 权重网格阶段，为 `corridor_l1` 的格子分配特殊约束，使其倾向于细长布局
- 或在评价阶段增加走廊长宽比检查项

【意见回复】：
- 加强走廊联通性评价指标，自然实现细长布局

### 1.3 文档3缺少 `dock1/dock2` 的平面约束描述

文档3第3.3节描述了 `dock2` 可在走廊内，但 `dock1` 在地面层的具体平面约束（是否需要在停车间内？）没有明确。当前代码 `NON_FUNCTIONAL = new Set(['corridor_l1', 'dock1', 'dock2'])` 将两者排除在效率计算之外，但吊装口的面积来源没有文档说明。

【意见回复】：
- docker1的位置应在泵房维护区。

## 二、约束生长算法（文档1）

### 2.1 邻接约束满足率低时重置代价高

当前做法：若邻接约束未满足，重置重试最多20次（文档3 Step 4）。但 M-01（水表房并排）和 M-02（变压器房相邻）是 MUST 约束，随机种子选取很可能连续失败。建议：
- 对 MUST 邻接对采用**先占法**：将邻接对视为一个组合种子，一次性在权重网格中选取两个相邻的初始位置，而非各自独立随机再检查

【意见回复】：
- 允许失败

### 2.2 进化策略中精英比例偏低

当前配比：每轮9个方案中2个交叉、7个随机。在方案池已经较优时，7/9的随机探索会频繁产生低分方案，拖慢收敛。建议：
- 动态调整配比：前 N 轮保持高随机比例（探索期），后期自动提高交叉比例（开发期）
- 例如：第1-3轮 2+7，第4-6轮 4+5，第7轮后 6+3

【意见回复】：
- 目前过快收敛了

### 2.3 种子位置交叉机制描述模糊

文档1说"种子位置坐标允许微小偏移"，但偏移量的定义不明确：
- 偏移是绝对格子数（如 ±2格）还是相对于建筑尺寸的比例？
- 偏移时是否需要重新满足权重约束？

建议在文档1补充交叉变异的精确定义。

【意见回复】：
- 建议种子位置可以在当前方案的房间内接矩形的中心重新定位

## 三、边界优化算法（文档2）

### 3.1 锯齿消除可能导致走廊断裂

边界二次调整允许"Give-and-Take"置换，但走廊（`corridor_l1`）是连通性关键节点。如果走廊与某房间的边界被平滑掉，可能导致：
- 走廊在某处收窄至0（拓扑上连通但宽度不满足1500mm）
- 走廊被切成两段
文档2的"拓扑连通性底线"只说"严禁切断房间"，但没有覆盖**走廊宽度退化**的情况。建议：
- 在阶段3执行中，对 `corridor_l1` 的格子增加最小宽度守护：任意横截面上走廊宽度不得低于3格（1500mm）

【意见回复】：
- 同意

### 3.2 评分系数 α/β 缺乏标定依据

文档2中 `Score = (α × 顶点收益) + (β × 利用率收益)`，但两个系数的取值没有说明。顶点收益（整数）与利用率收益（0~1的浮点数）量纲不同，α/β 的选取直接影响优化方向。建议：
- 在 `scorer-params.js` 中统一管理这两个系数，并在文档2中注明当前值及调参经验

【意见回复】：
- 在文档中标记：当前实现先忽略

### 3.3 面积波动容忍度方向性问题

文档2规定面积可在 `[95%, 110%]` 波动，上限110%意味着经过边界优化后房间可能超出用户设定的目标面积。对于 `parking`（最小面积=目标面积，150m²）和 `repair_zone`（最小120m²）这类有硬性下限的房间，超出反而是好事；但对于 `meter_main`（12m²）超出到13.2m²可能产生多余空间。建议：
- 对小型服务用房（`meter_main/meter_sub/fire_equip`）的波动上限收紧至 `105%`

## 四、评价模块（文档4）

### 4.1 基础分10000过高，遮蔽了差异

所有方案都从10000分开始，最大奖励约 +150+80+40+40+50+30=390分，最大惩罚（无violations时）约在几百分量级。这意味着评分区间集中在 `[9400, 10400]`，用户看到的方案差异往往只有几十分，**不直觉**。建议：
- 将基础分改为0，所有项都是加分和扣分
- 或将基础分降到100，让评分结果落在 `[0, ~200]` 这个更易感知的区间

### 4.2 `efficiency` 的计算用包围盒而非实际面积

当前 `floorFunctionalArea` 中用 `p.w * p.d`（包围盒面积）计算效率，但对于 L 形/U 形房间，包围盒面积会虚高实际占用。这会使包含复杂形状房间的方案效率虚高，掩盖形状质量问题。建议：
- 空间有效率应使用房间实际格子数（`cellCount × GRID_SIZE²`）而非包围盒
【意见回复】：
- 没有错，包围盒大，而实际格子少，这正是效率低的情况。

### 4.3 方案相似度惩罚预留为0但未实现

文档4的 `diversityPenalty` 注释为"预留为0"，但进化算法最容易产生的问题之一就是**方案高度相似**（前9名可能差异极小）。建议：
- 实现一个简单的相似度计算：以各房间质心坐标的欧氏距离作为方案间差异度，对相似度超阈值的方案施加 `−30pt` 惩罚
- 这可以直接提升用户看到的方案多样性
【意见回复】：同意

### 4.4 便捷性计算只考虑地面层入口

`computeAccessibilityScore` 的入口点固定为 `(buildingW/2, buildingD)`（建筑南侧中点），只对地面层7个房间计算距离，完全不考虑一层的交通便捷性。建议：
- 增加一层便捷性：以楼梯位置为参考点，计算一层各房间到楼梯的距离
- 两层各占50%权重合并为最终便捷性得分
【意见回复】：
- 在文档中标记：当前实现先忽略

## 五、优先级汇总

| 优先级 | 问题 | 影响 |
|--------|------|------|
| 高 | 走廊宽度约束无执行路径 | 生成方案可能不合规 |
| 高 | MUST 邻接对先占法 | 减少无效重试，提升生成成功率 |
| 高 | `efficiency` 用实际面积而非包围盒 | 评分失真 |
| 中 | 基础分改为0 | 用户体验，方案差异更直观 |
| 中 | 方案相似度惩罚实现 | 提升方案多样性 |
| 中 | 进化配比动态调整 | 加速收敛 |
| 低 | α/β 系数文档化 | 可维护性 |
| 低 | 一层便捷性纳入计算 | 评价完整性 |
